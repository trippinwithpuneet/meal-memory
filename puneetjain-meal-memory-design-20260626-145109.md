## Problem Statement

Self-cooking households — couples and small families who cook for themselves — experience a daily "decision fatigue" moment when choosing what to make. The problem isn't that they don't know how to cook. It's that in a fatigued state (sleepy morning, middle of a meeting), they can't retrieve the dishes they already know and love. The workaround today is memory, WhatsApp messages between partners, or defaulting to the same 3-4 "safe" fallbacks (egg sandwich, besan cheela). The gap is a retrieval and planning problem, not a discovery problem.

A secondary but adjacent pain exists in Indian households with domestic cooks: the decision-maker gets called mid-workday and is completely stumped. The WhatsApp/PDF export of a weekly plan is a natural adjacent feature for this audience, but is out of scope for V1.

## What Makes This Cool

The core insight: you don't need to discover new recipes, you need a system that surfaces your *own* loved recipes at the moment you're too tired to remember they exist. The recipe bank of dishes you've actually cooked and loved is the moat — not editorial curation, not algorithmic discovery.

The delight moment: on a Sunday evening, you open the app, see your 30-40 loved recipes as a grid, drag them into a Mon-Sun breakfast/lunch/dinner calendar, and you're done. Monday morning you wake up and the app tells you "You're having dal makhani tonight. Soak the kidney beans now." That's the whoa.

## Constraints

- Both household members have iPhones, separate Apple IDs → iOS-only for V1, with real-time household sync via Supabase Realtime
- At least one household member is gluten-free and dairy-free → dietary constraint modeling is a first-class concept, not an afterthought. Each recipe must be taggable with allergen/dietary flags (gluten-free, dairy-free, vegan, etc.). The weekly grid must visually warn if a recipe assigned to a slot conflicts with any household member's constraints.
- Open source project — distribution must work without requiring users to run their own server infrastructure
- V1 must be usable by the builder's own household within weeks, not months

## Premises

1. **Retrieval at the fatigue moment, not discovery**: The job is surfacing your own loved dishes when your brain can't, not finding new recipes. Discovery/external recipe suggestions ("feeling adventurous") are V2.
2. **Grid-based weekly slotting is intentional**: Research says rigid day-locked plans increase abandonment. This household consciously wants structure — the plan is meant to be a commitment, not a suggestion pool. This is a known tradeoff, accepted.
3. **V1 scope = recipe bank + manual weekly grid**: Everything else — grocery checklist, "feeling adventurous" Pinterest search, menstrual-cycle sync, nutrition mapping, prep reminders — is V2+. The narrowest usable slice is recipe bank + drag-into-grid.
4. **Target: self-cooking households**: The couple or small family who cook for themselves. Cook-briefing households are an adjacent market for later.
5. **Open source, iOS-native, minimal backend infrastructure**: Supabase cloud free tier handles household sync (no server to manage). App Store/TestFlight handles distribution. Open source users create a free Supabase account; self-hosters can run Supabase via Docker.

## Approaches Considered

### Approach A: Expo (React Native) + local-first SQLite (iOS target)
Cross-platform foundation, iOS-only build, local SQLite for recipe storage, iCloud Drive for sync. Large contributor pool.
- **Pros:** Large open source ecosystem, Expo EAS Build handles TestFlight without a Mac CI, cross-platform path remains open
- **Cons:** iCloud Drive sync is a hack (JSON files, no real-time, conflict edge cases), React Native UI never feels quite as native, requires API keys for photo-to-recipe extraction

### Approach B: Native SwiftUI + Supabase ✓ SELECTED
Native iOS app, Supabase for real-time household sync across separate Apple IDs, Vision framework for on-device recipe photo extraction.
- **Pros:** Supabase Realtime (websockets) gives instant sync between two phones with separate Apple IDs — the core non-negotiable requirement. Official Swift SDK (`supabase-swift`). Postgres data model is clean and queryable. Free tier covers a personal household easily. Open source users set up a free Supabase account; no Apple developer entanglement for sync. Vision framework on-device OCR means no API key for recipe photo import. Genuinely native iOS feel.
- **Cons:** Requires internet connection — no offline-first (acceptable for a planning app used on weekends). Supabase free tier pauses projects after 1 week of inactivity (fix: a trivial cron ping). Requires macOS/Xcode to build.

### Approach C: PWA (web-first)
Fastest iteration cycle, trivial self-hosting, but weaker fit for the sleepy-morning mobile use case. Not selected — the primary moment is on a phone in bed.

## Recommended Approach

**SwiftUI + Supabase.** Sync is the non-negotiable core feature for a two-person household with separate Apple IDs. CloudKit Sharing (the native Apple path) is the highest-risk approach for the most critical feature. Supabase Realtime delivers instant, websocket-based sync with a clean Swift SDK, zero server management, and a free tier that covers a personal household indefinitely. SwiftUI delivers the right native feel for a daily morning habit. Vision framework handles recipe photo extraction on-device, making the open source version dependency-light.

**Tech stack:**
- SwiftUI (iOS 17+)
- Supabase (cloud free tier) — Postgres for data model, Supabase Realtime for live sync, Supabase Auth for household member accounts
- `supabase-swift` (official Swift SDK)
- Vision framework (on-device OCR for photo-based recipe import — extracts raw text into an editable template; ingredient parsing is user-confirmed, not auto-inferred)
- URL recipe import — **deferred from V1 until Open Question #2 is resolved**; V1 users can paste a URL and enter ingredients manually
- TestFlight for beta distribution; App Store for eventual open source release

## Open Questions

1. **Supabase free tier inactivity pause**: Supabase pauses free-tier projects after 1 week of inactivity. Fix: a simple cron job (GitHub Actions on a schedule, or a free uptime monitor like UptimeRobot) that pings the Supabase health endpoint every few days. One-time setup, zero cost.

2. **Recipe import from URLs**: Web scraping recipes from Pinterest/food blogs is a browser problem (CORS, JavaScript-rendered pages). On iOS without a server, this likely requires WKWebView-based extraction or a community-maintained list of scrapers per domain. Scope this carefully before committing to it in V1.

3. **"Feeling adventurous" external suggestions**: Pinterest API requires developer account approval and has tight access restrictions. Google Custom Search or a curated recipe RSS feed may be more practical. This is V2 but the API choice should be decided early to avoid a design dead-end.

4. **Menstrual cycle sync**: Apple HealthKit provides cycle tracking data (HealthKit: HKCategoryTypeIdentifierMenstrualFlow). This is a clean integration point. Needs explicit user consent and careful privacy handling. V2+ scope confirmed.

5. **Nutrition mapping**: Nutritionix or USDA FoodData Central are the standard APIs. Both require keys. Could also start with a curated ingredient database. Scope: V2+.

## Success Criteria

- Puneet and his wife have their 30-40 loved recipes in the app and have used the weekly grid for 3 consecutive weeks without reverting to "what should we make?"
- Recipe entry via photo (Vision OCR → editable template) and manual entry both work without friction; URL import is a stretch goal
- Both iPhones see the same weekly plan in real time — an edit on one phone appears on the other within 2 seconds (Supabase Realtime)
- Dietary constraint flags work: a gluten-containing recipe assigned to the gluten-free member's slot shows a visible warning
- Empty/unplanned slots on the day of show a "nothing planned — pick now" prompt with a quick-access recipe list
- Recipes can be archived (soft-delete) without permanent loss; the bank stays curated without growing stale
- The app is open-sourced and at least one other household has used it via TestFlight

## Distribution Plan

- **TestFlight** for early household testing (invite-only, no App Store review needed)
- **GitHub** for open source — MIT license, contributor guide, Xcode setup instructions
- **App Store** once stable — free, open source, no monetization in V1
- **CI/CD**: Xcode Cloud (free tier for open source) or GitHub Actions with `fastlane match` for code signing automation

## Next Steps

1. **Set up Supabase project + define data model** — create a free Supabase project, design the Postgres schema (households, members, recipes, meal_slots), enable Realtime on the meal_slots table. Spike: two phones logged in with separate accounts, one edit on one phone appears instantly on the other. This should take one evening, not 2-3 weekends.
2. **Define the Recipe data model** — fields: name, photo_url, ingredients (jsonb array), steps (jsonb array), prep_time_mins, dietary_tags (array: gluten-free, dairy-free, vegan, etc.), archived flag, created_by. Lock this before building any UI.
3. **Build the recipe bank screen** — list of saved recipes, add/edit flow, photo capture with Vision OCR outputting into the editable template (user confirms ingredient parse, not auto-inferred). Photos stored in Supabase Storage.
4. **Build the weekly grid** — Mon-Sun × Breakfast/Lunch/Dinner, drag-and-drop from recipe bank into slots, swipe-to-clear, live sync via Supabase Realtime, dietary conflict warning on incompatible slots, "nothing planned — pick now" fallback for past-due empty slots.
5. **Add the planning-day reminder** — local notification on a user-configurable day (default Friday), tapping opens directly to the weekly planner for the upcoming week.
