## Vision

### 10x Check

The app knows everything you've cooked for the past 12 months. Meal history powers rotation nudges ("you've had this 4 times recently") and future auto-fill. Sunday planning takes 90 seconds: you pick from your recipe bank into the grid, your wife sees the update instantly, grocery list is auto-generated. Tuesday 9pm: "Soak kidney beans — dal makhani tomorrow." Friday: "Great week. 4/5 meals as planned." For 15 million Indian households with cooks, Sunday's plan exports to WhatsApp in one tap. That's the 10x product — and it's all additive on the V1 data model.

### Platonic Ideal

The app is not something you use. It's something that works while you live your life. Sunday at 7pm, open it. Your 30-40 loved recipes are right there. You drag a week together in 90 seconds. Wife sees the update instantly. Monday morning: "Tonight is dal makhani. Soak kidney beans now." An unplanned Tuesday: she opens the app, types what's in the fridge, three matching recipes surface immediately. It doesn't care that you're tired or that it's a Tuesday. It just works.

Note: V1 grid is manual drag-and-drop. Smart auto-fill based on rotation history is V2 — it needs meal_slots history, which V1 starts building now. Emergency Mode in V1 is reactive/manual (user-initiated), not proactive.

## Postgres Schema (V1 implemented + V2 reserved)

```sql
-- V1: implement these

CREATE TABLE households (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name text NOT NULL,
  timezone text NOT NULL DEFAULT 'Asia/Kolkata',
  created_at timestamptz DEFAULT now()
);

CREATE TABLE members (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  household_id uuid NOT NULL REFERENCES households ON DELETE CASCADE,
  user_id uuid NOT NULL,  -- Supabase auth.users.id
  name text NOT NULL,
  dietary_restrictions text[],     -- what they CANNOT eat: ['gluten-free', 'dairy-free']
  apns_device_tokens text[] DEFAULT '{}',  -- one per device; updated on every app launch
  created_at timestamptz DEFAULT now()
);

CREATE TABLE invite_tokens (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  household_id uuid NOT NULL REFERENCES households ON DELETE CASCADE,
  token text NOT NULL UNIQUE,       -- 6-char alphanumeric
  expires_at timestamptz NOT NULL,  -- 48h from creation
  used_at timestamptz               -- NULL = unused
);

CREATE TABLE recipes (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  household_id uuid NOT NULL REFERENCES households ON DELETE CASCADE,
  name text NOT NULL,
  photo_url text,
  ingredients jsonb,    -- [{name: text, quantity: text, unit: text}]
  steps jsonb,          -- [{text: text, hours_before: int}]
                        -- hours_before > 0 = advance prep (triggers alert)
                        -- hours_before = 0 = regular step
  prep_time_mins int,
  safe_for_tags text[], -- what recipe IS safe for: ['gluten-free', 'dairy-free']
  archived boolean DEFAULT false,
  created_by uuid REFERENCES members,
  created_at timestamptz DEFAULT now()
);

CREATE TABLE meal_slots (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  household_id uuid NOT NULL REFERENCES households ON DELETE CASCADE,
  week_start date NOT NULL,     -- Monday in household.timezone
  day_of_week int NOT NULL,     -- 0=Mon, 1=Tue, ..., 6=Sun
  meal_type text NOT NULL,      -- 'breakfast' | 'lunch' | 'dinner'
  recipe_id uuid REFERENCES recipes ON DELETE SET NULL,
  updated_at timestamptz NOT NULL DEFAULT now(),  -- set by CLIENT, not server default
  created_at timestamptz DEFAULT now(),
  UNIQUE (household_id, week_start, day_of_week, meal_type)
);

-- V2: reserved empty tables
CREATE TABLE grocery_items (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  household_id uuid REFERENCES households,
  recipe_id uuid REFERENCES recipes,
  ingredient_name text, quantity text, unit text,
  week_start date, checked boolean DEFAULT false
);
CREATE TABLE nutrition_cache (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  recipe_id uuid REFERENCES recipes,
  calories int, protein_g float, carbs_g float,
  fat_g float, iron_mg float, calcium_mg float,
  last_computed timestamptz
);
```

Enable Supabase Realtime on: `meal_slots` (V1), `grocery_items` (V2 when built).

### Row Level Security (RLS)

```sql
ALTER TABLE households ENABLE ROW LEVEL SECURITY;
ALTER TABLE members ENABLE ROW LEVEL SECURITY;
ALTER TABLE recipes ENABLE ROW LEVEL SECURITY;
ALTER TABLE meal_slots ENABLE ROW LEVEL SECURITY;
ALTER TABLE invite_tokens ENABLE ROW LEVEL SECURITY;

-- Household access via membership
CREATE POLICY "household access via membership"
  ON households FOR ALL
  USING (id IN (SELECT household_id FROM members WHERE user_id = auth.uid()));

CREATE POLICY "members can access own household"
  ON members FOR ALL
  USING (user_id = auth.uid());

CREATE POLICY "recipes scoped to household"
  ON recipes FOR ALL
  USING (household_id IN (SELECT household_id FROM members WHERE user_id = auth.uid()));

CREATE POLICY "meal_slots scoped to household"
  ON meal_slots FOR ALL
  USING (household_id IN (SELECT household_id FROM members WHERE user_id = auth.uid()));

-- invite_tokens: three policies
CREATE POLICY "invite tokens readable to join"
  ON invite_tokens FOR SELECT
  USING (expires_at > now() AND used_at IS NULL);

CREATE POLICY "members can create invite tokens"
  ON invite_tokens FOR INSERT
  WITH CHECK (household_id IN (SELECT household_id FROM members WHERE user_id = auth.uid()));

CREATE POLICY "anyone can claim unused token"
  ON invite_tokens FOR UPDATE
  USING (used_at IS NULL AND expires_at > now())
  WITH CHECK (used_at IS NOT NULL);  -- can only set used_at, not unset it
```

**Realtime + RLS (critical step):** After enabling RLS, go to Supabase Dashboard → Database → Replication and enable **Row Level Security** for the `meal_slots` table. This is separate from `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`. Without it, all households receive each other's slot change events. The `supabase-swift` SDK passes the user's JWT automatically when using `supabase.auth` — no extra code needed, just verify during the sync spike.

## Engineering Notes

### Auth Strategy (Resolved)
Required auth from day 1. Email/password for V1 (fastest). Add Sign in with Apple before App Store submission (Apple requires it when email/password is offered on iOS).

### Household Invite Flow
1. Creator: gets 6-char invite code + QR on household creation. Token stored in `invite_tokens` with 48h expiry.
2. Joiner: signs up → enters code → app queries `invite_tokens WHERE token = input AND expires_at > now() AND used_at IS NULL` → creates `members` row → marks token `used_at = now()`.
3. Both phones now share `household_id`. Realtime subscription starts.

### OCR: Heuristic Ingredient + Step Extraction
Run `VNRecognizeTextRequest` on background queue (`DispatchQueue.global(qos: .userInitiated)`). On completion, split recognized lines heuristically:
- Lines matching a quantity pattern (`^\d[\d\/\s]*(cup|tbsp|tsp|kg|g|ml|l|pinch)?\s+.+`) → pre-populate as ingredients
- Remaining lines → pre-populate as steps (with `hours_before: 0`)

Show the pre-populated form for user review and correction. "Clear OCR / start over" button always visible. Falls back to blank manual form if OCR returns garbage.

This gives real value: a recipe photo becomes a mostly-filled form in 2 seconds. User corrects mistakes rather than transcribing from scratch.

### URL Recipe Import (V1 — moved from V2 during design review)

Mobile browsers block cross-origin fetches to arbitrary URLs (CORS). A Supabase Edge Function acts as a server-side proxy:

```typescript
// supabase/functions/fetch-recipe/index.ts (Deno)
// POST { url: string } → { name, ingredients, steps, prepTimeMins }

import { DOMParser } from "https://deno.land/x/deno_dom/deno-dom-wasm.ts"

serve(async (req) => {
  const { url } = await req.json()

  // Validate: must be http/https, not an internal IP
  const parsed = new URL(url)
  if (!['http:', 'https:'].includes(parsed.protocol)) {
    return new Response(JSON.stringify({ error: 'Invalid URL' }), { status: 400 })
  }

  const html = await fetch(url, {
    headers: { 'User-Agent': 'Mozilla/5.0 (compatible; MealMemory/1.0)' }
  }).then(r => r.text())

  // Strategy 1: JSON-LD structured data (most recipe blogs use schema.org/Recipe)
  const doc = new DOMParser().parseFromString(html, 'text/html')
  const jsonLdScripts = doc.querySelectorAll('script[type="application/ld+json"]')
  for (const script of jsonLdScripts) {
    try {
      const data = JSON.parse(script.textContent ?? '')
      const recipe = Array.isArray(data) ? data.find(d => d['@type'] === 'Recipe') : 
                     data['@type'] === 'Recipe' ? data : 
                     data['@graph']?.find((d: any) => d['@type'] === 'Recipe')
      if (recipe) return Response.json(parseJsonLdRecipe(recipe))
    } catch {}
  }

  // Strategy 2: OpenGraph / meta tags (fallback — gives name only)
  const ogTitle = doc.querySelector('meta[property="og:title"]')?.getAttribute('content')
  if (ogTitle) return Response.json({ name: ogTitle, ingredients: [], steps: [] })

  return new Response(JSON.stringify({ error: 'No recipe found' }), { status: 422 })
})

function parseJsonLdRecipe(r: any) {
  const ingredients = (r.recipeIngredient ?? []).map((line: string) => {
    // Parse "2 cups whole milk" → {quantity: "2 cups", name: "whole milk"}
    const match = line.match(/^([\d\s\/]+\s+\w+)\s+(.+)$/)
    return match ? { quantity: match[1].trim(), name: match[2].trim(), unit: '' }
                 : { quantity: '', name: line.trim(), unit: '' }
  })
  const steps = (r.recipeInstructions ?? []).map((s: any, i: number) => ({
    text: typeof s === 'string' ? s : (s.text ?? ''),
    hours_before: 0
  }))
  return {
    name: r.name ?? '',
    ingredients,
    steps,
    prepTimeMins: parseDuration(r.prepTime),
  }
}
```

**What this covers:**
- Most food blogs (Serious Eats, Smitten Kitchen, VegRecipesOfIndia, etc.) use `schema.org/Recipe` JSON-LD — Strategy 1 succeeds ~80% of the time
- Pinterest board pins link to the source blog, not Pinterest-hosted recipes; the iOS app should resolve the final URL before sending (follow redirects)
- For unstructured pages (Strategy 1 fails), the form pre-populates with just the recipe name — user fills in the rest manually, which is still better than starting from scratch

**Edge Function security:** validate URL is public HTTP/HTTPS before fetching; Edge Function runs with Supabase service key, so rate-limit by household (1 import per 5 seconds) to prevent abuse. iOS app can call it with the user's JWT for auth.

### Prep Alert Architecture (Chosen: Supabase Edge Function + Custom APNs JWT)

```typescript
// supabase/functions/prep-alert/index.ts (Deno)
// Runs at 9pm daily via Supabase CRON

const teamId = Deno.env.get('APNS_TEAM_ID')!
const keyId = Deno.env.get('APNS_KEY_ID')!
const apnsKey = Deno.env.get('APNS_PRIVATE_KEY')!  // .p8 contents

// Generate APNs JWT (ES256) — fresh per request, expires in 60 min
const apnsToken = await generateAPNsJWT(apnsKey, keyId, teamId)

// Query tomorrow's slots with advance prep steps
const tomorrow = new Date(); tomorrow.setDate(tomorrow.getDate() + 1)
const { data: slots } = await supabase
  .from('meal_slots')
  .select('recipe_id, household_id, recipes(name, steps)')
  .eq('week_start', getMondayFor(tomorrow))
  .eq('day_of_week', getDayOfWeek(tomorrow))

for (const slot of slots ?? []) {
  const advancePrepSteps = slot.recipes.steps
    .filter((s: Step) => s.hours_before > 0)
  if (!advancePrepSteps.length) continue

  const alertText = `${slot.recipes.name} tomorrow — ${advancePrepSteps[0].text} tonight.`

  // Get all device tokens for this household
  const { data: members } = await supabase
    .from('members')
    .select('apns_device_tokens')
    .eq('household_id', slot.household_id)

  for (const token of members?.flatMap(m => m.apns_device_tokens) ?? []) {
    const resp = await fetch(`https://api.push.apple.com/3/device/${token}`, {
      method: 'POST',
      headers: {
        'authorization': `bearer ${apnsToken}`,
        'apns-topic': 'com.yourdomain.mealmemory',
        'apns-push-type': 'alert',
      },
      body: JSON.stringify({ aps: { alert: alertText, sound: 'default' } })
    })
    if (resp.status === 410) {
      // Stale token — remove from members.apns_device_tokens
      await supabase.rpc('remove_apns_token', { token, household_id: slot.household_id })
    }
  }
}
```

APNs JWT is generated fresh per Edge Function invocation (Apple's JWT-based APNs requires a fresh token per request anyway — no long-lived certs to rotate).

### Upsert: Client-side `updated_at` (critical for offline correctness)
Every `meal_slots` upsert must include `updated_at: Date()` from the **device clock**, not rely on `DEFAULT now()` on the server. If the server sets the timestamp, an offline edit gets the reconnection time (always newest), silently clobbering the other person's changes. With device timestamp, the last actual edit wins regardless of network timing.

```swift
// Every slot write
let slotData: [String: AnyJSON] = [
  "household_id": householdId,
  "week_start": weekStart,
  "day_of_week": dayOfWeek,
  "meal_type": mealType,
  "recipe_id": recipeId,
  "updated_at": ISO8601DateFormatter().string(from: Date())  // device clock
]
try await supabase.from("meal_slots")
  .upsert(slotData, onConflict: "household_id,week_start,day_of_week,meal_type")
  .execute()
```

### Emergency Mode Search
Local, no backend call. On app launch, cache all non-archived recipes in memory.
1. User types "paneer spinach tomatoes"
2. Parse into tokens (lowercased, split on whitespace)
3. For each recipe: count ingredient name matches against tokens
4. Rank by match count descending; show top 5 with match count

### Conflict Resolution (Empty Slot vs Emergency Mode)
- **Empty slot fallback** — triggered automatically when current meal slot has no recipe. Calendar-aware, time-aware. Shows "Nothing planned — pick now" banner.
- **Emergency mode** — manually invoked anytime. User types what's in the fridge. No calendar context.

### Dietary Conflict Warning Logic
```swift
// For each slot assignment:
func hasConflict(recipe: Recipe, members: [Member]) -> Bool {
  members.contains { member in
    member.dietaryRestrictions.contains { restriction in
      !recipe.safeForTags.contains(restriction)
    }
  }
}
// member.dietaryRestrictions = what they CAN'T eat
// recipe.safeForTags = what the recipe IS safe for
// Conflict = member has restriction that recipe doesn't accommodate
```

|------|-------------------|----------|
| Realtime WebSocket disconnect | Silent desync, stale grid | TODO (Weekend 2): reconnect banner + REST re-fetch |
| Supabase upsert network error | Optimistic update not confirmed | Yes: error toast + revert |
| APNs device token stale after reinstall | Push notifications stop | TODO (Weekend 3): refresh on launch + 410 cleanup |
| Vision OCR poor quality photo | Garbage heuristic extraction | Yes: "Clear OCR / start over" button |
| Invite token race (two users claim same) | Second user rejected | Yes: `WHERE used_at IS NULL` prevents double-claim |
| Supabase free tier pause | App appears broken | Yes: GitHub Actions ping every 3 days |
